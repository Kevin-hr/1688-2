# 1688 to Ozon Global - MVP 交付与执行规划

**文档版本**: v1.0
**角色**: 资深产品经理与技术架构师 (Senior Product Manager & Technical Architect)

---

## 1. 评估开发周期

基于当前中小型数据采集与接口对接类 MVP 的复杂度，我将本项目（1688 to Ozon 一键上架工具）的总开发与验证周期评估为 **2 周（1个敏捷冲刺 / 约 10-14 个工作日）**。

**评估依据（时间占比）：**
1. **需求评估与架构设计 (10%)**：1-2 天。明确 Ozon Global 的 API Schema 要求（`/v3/product/import`）以及 1688 的反爬机制与 DOM 结构解析。
2. **核心采集功能开发 (35%)**：4 天。实现 Playwright 的无头单例多并发采集、基础的图文数据解析、基于 CSS 选择器的智能 fallback 抓取。
3. **数据清洗与 Ozon 规则转换 (25%)**：3 天。研发 `OzonTransformer`，完成标题清洗、单位自动转换（kg→g, cm→mm），以及免费翻译引擎的对碰集成（中转俄/英）。
4. **接口集成与对接 (15%)**：2 天。编写独立上架脚本 `upload_to_ozon.py`，完成 API 鉴权、JSON Payload 构造以及批量提交保护（100条/次）。
5. **内部测试与联调交付 (15%)**：2-3 天。执行端到端（E2E）测试，输出完整的安装文档（README、测试样例），以及最后交付阶段的验收支持。

## 2. 制定 MVP 交付计划

为了确保交付的可控性和可落地性，建议采取如下分阶段验收计划：

| 阶段 | 核心任务 | 里程碑与验收物 | 预期时间节点 |
| :--- | :--- | :--- | :--- |
| **Stage 1 (需求与架构)** | Ozon 字段格式要求映射、仓库初始化、业务流梳理。 | 输出《架构设计与数据流向图》、确立开发依赖包 | Day 2 结束 |
| **Stage 2 (核心采集实现)** | 1688 核心爬虫逻辑编写（支持 URL 单采、关键字批量搜和翻页），以及多线程图片下载。 | `$ python main.py` 可稳定输出 `detail.md` 和原始全量 `raw.json` | Day 6 结束 |

## 3. 梳理已验证的核心功能清单

本着“极简验证，击穿痛点”的最小化 MVP 原则，我们确认保留以下**核心功能**，并暂缓非阻碍性的高级需求：

### ✅ **已验证可行的 MVP 核心功能：**
* **自然语言级触发路由 (`task_router.py`)**：支持从终端通过关键字（例：`python main.py 猫头鹰玩具`）或具体的 `1688.com/offer/xxx.html` 直连采集。
* **单浏览器上下文复用 (Session Reuse)**：通过异步 `async with` 长链接采集，规避了启动/关闭浏览器的性能开销，已验证对反爬阻力最低。
* **Ozon 原生化清洗引擎 (`OzonTransformer`)**：验证了提取“重量/尺寸”并标准化（KG→G，CM→MM），并去除了“厂家直销”等无用修饰词。并附带自动容错降级的 **Google/Bing 中转俄翻译系统**。
* **API 一键上架管道工具 (`upload_to_ozon.py`)**：验证了只需 `.env` 中提供 `Client-Id` 和 `Api_Key`，即可脱离繁琐后台打通 `/v3/product/import` 接口的闭环流程。

### ⛔ **MVP 阶段已明确排除的复杂/非核心功能：**
* **全自动化类目映射字典 (NLP Category Matching)**：目前采取统一传入默认 ID 的占位符方案，规避了巨大的类目数据清洗成本。
* **逆向工程与破验证码 (Anti-Captcha/WAF Bypass)**：由于属于初期测试，MVP 不集成高昂 IP 代理池池和识别打码平台，全凭本地自然环境发包。
* **打点后的订单/库存自动同步 (Stock & Order Sync)**：MVP 仅验证“上架”这一单边推进行为。
* **图形可视化配置界面 (GUI / Web Dashboard)**：仅保留简洁专业的 CLI 命令行供执行与调试。

> **架构师谏言**：上述 MVP 清单足以向客户证明**系统打通 1688 数据孤岛到 Ozon 的端到端链路的可行性**。建议客户拿到 Release v1.3.0 后立刻测试使用 3-5 款不敏感爆款商品测试推流，待后台 Draft (草稿) 中能识别到数据，再谈后续二期重构（比如类目匹配、云端代理）。
